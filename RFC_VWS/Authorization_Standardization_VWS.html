Authorization Standardization

Dutch Healthcare Profile (AuthZEN-based)

Summary

Within the Dutch healthcare domain, authorization is positioned as a generieke functie (generic function).

  • operate system-wide across healthcare domains
  • be independent of individual applications
  • be standardizable and normable
  • support interoperability

This RFC sits at the third level of a three-tier specification hierarchy. Each level inherits from and extends the level above it — lower levels may only add constraints, not contradict the levels above.

Architecture Overview

This RFC is PDP-agnostic: it standardizes the PEP–PDP interaction at the protocol level, without prescribing how policies are authored or evaluated. Any engine exposing an AuthZEN-compliant decision endpoint — ABAC, RBAC, ReBAC, or hybrid, built on OPA, Cedar, OpenFGA, XACML, or a custom engine — can be plugged in without requiring changes to consuming applications, allowing organizations to choose their PDP while preserving a uniform authorization interface.

The Preprocessor has no authorization intelligence of its own. Its sole responsibility is to translate an incoming request into a well-formed AuthZEN authorization request — populating the subject, action, resource, and context fields — so that the PDP can interpret and evaluate it. All policy logic resides exclusively in the PDP.

This RFC serves as the parent contract for authorization standardization within Dutch healthcare (Informatiestandaarden Zorg). It establishes the generic, domain-neutral base structure for AuthZEN-based authorization decisions — applicable across all healthcare domains including iWlz, Zvw, Wmo, and Jeugdwet. Domain-specific RFCs inherit from this document and extend it by defining which attributes must be present in the AuthZEN request and how they are populated within that domain's context. Domain RFCs must not deviate from the base structure defined here; they may only add domain-specific constraints or attribute requirements on top of it. For the applicable scope of each domain, refer to the corresponding domain RFC.

Scope

This RFC defines the generic authorization contract for the Dutch healthcare domain. It is the normative parent specification. Domain-specific RFCs inherit this contract and may only extend it — they may not deviate from or replace the base contract defined here.

The following domains fall within scope:

Domain Description Source
iWlz Information provision for long-term care (Wlz). Authorization governs which organizations may access or mutate client care records based on their role (zorgkantoor, zorgaanbieder, CIZ, CAK) and regional or contractual relationship. iWlz RFC repository
Zvw Basic health insurance (Zorgverzekeringswet). Authorization governs access to declaratie, vergoeding, and verzekeringsdata based on the insurer–provider relationship and population ownership. Vektis standaardisatie
Wmo Social support act (Wet maatschappelijke ondersteuning 2015). Authorization governs municipality access to citizen support decisions and provider access to iWmo messages within active municipal contracts. iStandaarden iWmo
Jeugdwet Youth care act. Authorization governs municipality access to youth care decisions and youth care provider access to iJw messages within active assignments. iStandaarden iJw

Out of scope

  • Authentication and identity proofing — governed by NLGov OAuth/OIDC profiles.
  • Policy authoring and evaluation logic inside the PDP — only the PEP–PDP interface is standardized here.
  • Patient/citizen consent management — addressed separately under MITZ/NUTS.

Relation to Other Generic Functions

Authorization is one of several generieke functies (generic functions) defined under the VWS program for Dutch healthcare interoperability. These functions are designed to be composable: each addresses a distinct concern, and together they form the infrastructure layer for trusted data exchange.

This RFC covers the authorization generic function only. The table below positions it relative to the adjacent generic functions it depends on or interacts with.

Generic Function Concern Relation to this RFC Reference
Care Services Directory (LRZa) Addressing and discovery — publishing, finding, and resolving trusted addressing information for organizations, services, locations, and endpoints. LRZa is the authoritative source for the subject.id (organization identifier) used in AuthZEN requests. The OrganizationAffiliation resource in LRZa models inter-organizational relationships (e.g. a zorgaanbieder acting within a zorgkantoor region) that are candidate context attributes for authorization decisions. The PEP resolves subject identity via LRZa before constructing the AuthZEN request. Care Services Directory specification
Authenticatie (NLGov OAuth/OIDC) Identity proofing and authentication — establishing who the calling party is before any authorization decision is made. Authentication is a prerequisite for authorization but is explicitly out of scope for this RFC. The PEP is responsible for authenticating the caller (via NLGov OAuth 2.0 / OIDC and PKIoverheid mTLS) before forwarding an AuthZEN request to the PDP. The authenticated identity is then mapped to the subject field. NLGov OAuth/OIDC profile
Toestemming / Consent (MITZ/NUTS) Patient and citizen consent management — recording and enforcing explicit consent for data access. Consent outcomes may be surfaced as context attributes in an AuthZEN request, but consent management itself is out of scope for this RFC. Domain-specific RFCs define whether and how consent attributes are included. MITZ / NUTS consent framework

End-to-end flow across generic functions

A typical cross-domain authorization check proceeds as follows:

  1. The calling organization authenticates using PKIoverheid mTLS and NLGov OAuth/OIDC (Authenticatie).
  2. The PEP resolves the caller's organization identity and any relevant affiliation or endpoint data from the LRZa Care Services Directory (Adressering).
  3. The Preprocessor constructs a well-formed AuthZEN request, populating subject, action, resource, and context from the above sources, following the contract defined in this RFC (Autorisatie).
  4. The PDP evaluates the request against domain policy and returns a decision.
  5. The PEP enforces the decision.

Authorization Contract

This contract inherits all normative rules from the NLGov AuthZEN profile and adds Dutch healthcare domain constraints on top. Any rule marked MUST, SHALL, or SHOULD in the NLGov profile applies here without repetition. The sections below document the full inherited field structure and the additional constraints this RFC introduces.

Every attribute included in an AuthZEN request must be traceable to an authoritative source and governed within a defined trust framework. Domain-specific RFCs specify which attributes are required, where they originate (e.g. LRZa, the calling organization's token, a consent registry), and who is responsible for keeping them accurate.

A well-formed AuthZEN access evaluation request has the following top-level structure:


POST /access/v1/evaluation
Content-Type: application/json
X-Request-ID: <opaque-id>

{
  "subject":  { ... },
  "action":   { ... },
  "resource": { ... },
  "context":  { ... }
}

The endpoint path SHOULD include the v1 version identifier as shown above, per the NLGov profile.

Subject

The subject identifies the party requesting access. In Dutch healthcare this is always an organization or system, not an individual end-user.

FieldRequiredTypeRule (inherited from NLGov)
typeREQUIREDstring Linked Data URI RECOMMENDED (e.g. https://www.w3.org/ns/org#Organization).
idREQUIREDstring Unique identifier scoped to type. MUST resolve to an entry in the LRZa Care Services Directory for Dutch healthcare subjects.
propertiesOPTIONALobject Additional subject attributes (organization type, roles, region). Domain RFCs define which properties are required within their scope. 

{
  "subject": {
    "type": "https://www.w3.org/ns/org#Organization",
    "id": "urn:oid:2.16.528.1.1007.3.3.12345",
    "properties": {
      "organization_type": "ZORGVERZEKERAAR",
      "roles": ["REQUESTER"]
    }
  }
}

Context

The context carries additional attributes that inform the authorization decision without being the subject, action, or resource. The NLGov profile defines a set of standard context fields; domain RFCs may add domain-specific attributes.

FieldRequiredTypeRule (inherited from NLGov)
timeOPTIONALstring RFC 3339 timestamp. Use for temporal authorization queries (e.g. validity at a specific point in time).
traceparentOPTIONALstring W3C Trace Context traceparent value (spec §3.2.2). RECOMMENDED for distributed tracing and audit.
tracestateOPTIONALstring W3C Trace Context tracestate value (spec §3.3.1.1).
mimOPTIONALURL URL to the applicable meta-information model (MIM standard). RECOMMENDED when the request involves structured information models.
ld-contextOPTIONALURL or object JSON-LD context mapping for semantic integration. RECOMMENDED per the NLGov profile for interoperable attribute semantics.

Domain RFCs define additional context attributes such as organizational affiliation, contractual region, or consent status.

Transport and Security

The following requirements are inherited directly from the NLGov AuthZEN profile and apply without exception to all implementations conforming to this RFC.

RequirementLevelDetail
Transport encryptionMUST All PEP–PDP communication MUST use TLS. Unencrypted HTTP is not permitted.
PEP authenticationSHOULD The PDP SHOULD authenticate the PEP using mutual TLS (mTLS with PKIoverheid certificates), OAuth 2.0 client credentials, or an API key.
Content typeMUST Requests and responses MUST use Content-Type: application/json.
JSON conformanceSHOULD JSON payloads SHOULD conform to the I-JSON profile (RFC 7493): UTF-8 encoding, no duplicate keys, no numeric precision loss.
X-Request-ID headerRECOMMENDED PEP SHOULD include an opaque X-Request-ID header; the PDP MUST echo it in the response if present.
HTTP status codesMUST 200 for success; 400 for malformed request; 401 for authentication failure (PDP SHOULD include WWW-Authenticate); 403 for authorisation failure; 500 for internal errors.
Rate limitingRECOMMENDED PDPs SHOULD implement rate limiting to protect against denial-of-service.
Discovery endpointSHOULD PDP SHOULD expose metadata at /.well-known/authzen-configuration including at minimum policy_decision_point and access_evaluation_endpoint.
Response signingMAY PDP MAY sign responses as JWS. Null-valued properties SHOULD be omitted.
